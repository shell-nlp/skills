# 数据库表设计

## 概述

将用户需求转化为面向生产环境的关系型表结构方案。优先兼顾正确性、查询性能、数据一致性、可演进性与运维简单性，并最终输出一份自包含的 Markdown 文档。

## 工作流

1. 提炼真实需求。
- 识别业务实体、生命周期、读写路径、核心筛选与排序条件、预估规模、保留策略、租户模型、一致性要求、审计或历史需求以及系统边界。
- 明确记录缺失信息。只有当缺失信息会实质性影响表结构时才追问；否则基于显式假设继续设计。
- 如果未指定数据库引擎，默认按 PostgreSQL 兼容的关系型 OLTP 设计处理，除非上下文明显更适合 MySQL 或其他引擎。

2. 选择建模策略。
- 围绕稳定的业务实体，优先从规范化模型开始。
- 只有在明确存在热点查询、报表投影或运维约束时，才引入反范式设计。
- 当保留策略、审计要求或写入模式不同，应将当前态表与追加写的历史表或事件表拆分。
- 多对多关系使用中间表建模。
- 对受控枚举值，在“受限枚举”和“字典表”之间做选择；当枚举值需要元数据或可能频繁变化时，优先字典表。
- JSON 或 JSONB 仅用于稀疏扩展字段，不用于核心关联、强约束或高频筛选条件。

3. 有意识地设计每一张表。
- 为每张表写一句明确用途说明。
- 明确字段类型、是否可空、默认值、单位和命名方式。
- 除非自然主键稳定、紧凑且对外有明确业务含义，否则优先使用代理主键。
- 仅为真实业务不变量添加唯一约束。
- 金额必须使用精确类型，推荐最小货币单位整数或精确数值类型，禁止浮点。
- 时间字段统一按 UTC 处理；若引擎支持，优先使用带时区语义的时间类型。
- 状态字段、软删除、版本号、审计字段只在确有生命周期或治理需求时引入。

4. 根据访问路径设计约束与运维规则。
- 索引必须从真实查询路径、关联方式、筛选条件、唯一性校验和排序需求推导出来。
- 优先少量高价值的复合索引，而不是大量互相重叠的索引。
- 只有在数据规模、保留周期或维护成本足以支撑复杂度时，才建议分区、归档或分片。
- 默认推荐外键；如果不使用外键，必须说明原因以及对应的补偿控制方案，例如应用层校验、CDC 校验、异步修复任务等。

5. 输出最终 Markdown。
- 生成最终答案前，读取 `references/output-template.md`，除非用户要求其他格式，否则按其章节顺序输出。
- 定稿前，读取 `references/design-checklist.md`，补查一致性、规模和生命周期遗漏。
- 只有在能显著提升说明效果，或用户明确要求可执行 SQL 时，才附带示例 DDL。

## 决策规则

- 每个业务事实尽量只保留一个事实来源；若需要冗余可变数据，必须说明同步与失败处理策略。
- 可空字段必须是有意设计；`NULL` 只能表达“未知”或“不适用”，不能拿来掩盖建模不清。
- 当分析型需求与事务型需求发生冲突时，应将分析模型与事务表拆开设计。
- 关系型多对多数据必须使用中间表，不能使用逗号分隔 ID、嵌套列表或不可约束的结构替代。
- `deleted_at` 不是默认项，而是产品需求项；如果没有恢复和审计要求，硬删除通常更简单、更安全。
- 当命名规范、排序规则、字符集或数据库特性会显著影响可移植性或正确性时，必须明确说明。

## 输出标准

- 最终只输出一份 Markdown 文档。
- 必须给出主推荐方案，不能只罗列多个未排序选项。
- 假设条件必须在文档前部显式列出。
- 文档至少覆盖：需求摘要、方案概览、逐表设计、索引与约束、生命周期与保留策略、必要时的示例 DDL、方案取舍与待确认问题。
- 结果既要让工程人员可落地，也要让产品或业务方能读懂。
- 输出语言跟随用户；对于中文需求，用中文写文档，同时保留 SQL 标识符原始大小写。
